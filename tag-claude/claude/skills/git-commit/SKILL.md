# Git Commit Skill

Create a git commit that follows the conventional commit specification with AI attribution tagging.

## Process

1. Run `git status` to review staged and unstaged changes. Never use the `-uall` flag.
2. Run `git diff --cached` to inspect staged changes. If nothing is staged, review unstaged changes and stage coherent, logically related changes to form an atomic commit. If you are unsure what belongs together, ask the user before staging anything.
3. Run `git log --oneline -5` to review recent commit style for consistency.
4. Compose the commit message following the format and rules below.
5. Present the draft commit message to the user for approval before committing. If the user requests changes, revise and re-present before committing.
6. Create the commit. Use a HEREDOC to pass the message to `git commit -m`. Never use `--no-verify`.

If `$ARGUMENTS` is provided, use it as a hint for the commit description but still follow all formatting rules.

## Commit Message Format

```
<type>(<optional scope>): [AI-tag] <description>

<optional body>

<optional footer(s)>
Co-Authored-By: Claude <noreply@anthropic.com>
AI-assisted: <what AI did and what the human did>
```

## Types

Use **only** these types:

| Type       | When to use                                             |
| ---------- | ------------------------------------------------------- |
| `feat`     | New feature (correlates with MINOR in semver)           |
| `fix`      | Bug fix (correlates with PATCH in semver)               |
| `docs`     | Documentation only changes                              |
| `style`    | Changes that do not affect the meaning of the code      |
| `refactor` | Code change that neither fixes a bug nor adds a feature |
| `test`     | Adding or correcting tests                              |
| `build`    | Changes to build systems or external dependencies       |
| `data`     | Adding or changing data included in the repository      |
| `revert`   | Reverting a previous commit                             |
| `chore`    | Changes that do not modify src or test files            |

## Scope

A scope MAY be provided after the type. It MUST be a noun describing the section of the codebase, surrounded by parentheses.

```
fix(parser):
feat(auth):
```

## Breaking Changes

Indicate breaking changes by appending `!` before the colon:

```
feat(api)!: [AI] redesign authentication endpoint
```

If `!` is used, `BREAKING CHANGE:` MAY be omitted from the footer. Otherwise, include `BREAKING CHANGE: <description>` as a footer.

## Description

- MUST immediately follow the colon and space after the type/scope prefix
- Aim for 80 characters or less for the full header line; 100 characters is the hard maximum (the AI tag may consume some of this budget)
- Use the imperative mood, present tense ("add feature" not "added feature")
- Do not capitalize the first letter
- Do not end with a period

## Body

- Begins one blank line after the description
- Free-form text, may consist of multiple paragraphs
- Provide additional context about the code changes

## Footer

- Begins one blank line after the body
- Each footer uses the format: `Token: value` or `Token #value`
- Footer tokens use hyphens instead of spaces (except `BREAKING CHANGE`)

## AI Attribution Tagging

Every commit MUST include an AI attribution tag in the description line and an `AI-assisted` trailer in the footer.

### AI Tags (in the description)

Place the tag **after** the colon-space and **before** the description text:

| Tag           | When to use                                                                                      |
| ------------- | ------------------------------------------------------------------------------------------------ |
| `[AI]`        | AI generated the substantial or core work (complete functions, features, algorithms, core logic) |
| `[AI-minor]`  | AI made minor contributions (formatting, boilerplate, small fixes, tab-completion-level help)    |
| `[AI-review]` | A human wrote the code; AI reviewed, suggested improvements, or assisted with debugging          |

### AI-assisted Trailer (in the footer)

Always include a `Co-Authored-By: Claude <noreply@anthropic.com>` trailer followed by an `AI-assisted:` line as the last footer explaining the division of work. Be specific and honest.

Examples:

- `AI-assisted: Core logic generated, tests human-written.`
- `AI-assisted: Full implementation generated by Claude, reviewed and edited by human.`
- `AI-assisted: Human-written code, AI suggested error handling improvements.`
- `AI-assisted: Boilerplate and type definitions generated, business logic human-written.`
- `AI-assisted: AI reviewed PR and suggested refactor, human implemented changes.`

## Complete Examples

### Feature with substantial AI contribution

```
feat(parser): [AI] add recursive JSON parser for nested objects

Implements a recursive descent parser that handles arbitrarily
nested JSON objects with proper error recovery and position
tracking.

Co-Authored-By: Claude <noreply@anthropic.com>
AI-assisted: Core parsing logic generated by Claude, edge case
tests and error messages written by human.
```

### Bug fix with minor AI help

```
fix(auth): [AI-minor] correct token refresh race condition

Adds mutex lock around the token refresh flow to prevent
concurrent refresh requests from invalidating each other.

Closes #247
Co-Authored-By: Claude <noreply@anthropic.com>
AI-assisted: Human diagnosed and fixed the bug, AI helped with
mutex syntax.
```

### Human-written code reviewed by AI

```
refactor(database): [AI-review] extract connection pooling into shared module

Moves connection pool logic from three separate services into a
shared database module to reduce duplication and standardize
connection handling.

Co-Authored-By: Claude <noreply@anthropic.com>
AI-assisted: Human wrote the refactor, AI reviewed for
correctness and suggested naming improvements.
```

### Breaking change with AI contribution

```
feat(api)!: [AI] redesign authentication to use OAuth 2.0

Replaces the custom token-based auth system with standard
OAuth 2.0 authorization code flow. All API endpoints now
require Bearer tokens obtained through the /oauth/token
endpoint.

BREAKING CHANGE: The X-Auth-Token header is no longer accepted.
All clients must migrate to OAuth 2.0 Bearer tokens.
Co-Authored-By: Claude <noreply@anthropic.com>
AI-assisted: Full implementation generated by Claude, security
review and configuration done by human.
```

### Docs change

```
docs(readme): [AI-minor] update installation instructions for v2

Co-Authored-By: Claude <noreply@anthropic.com>
AI-assisted: AI drafted updated instructions, human verified
accuracy and added platform-specific notes.
```
